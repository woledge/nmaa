## investment_club

Full-featured investment club management system.

**Author:** Ibrahim Elmasry
**Dependencies:** `base`, `mail`, `account`, `sale_management`, `product`, `portal`, `crm`, `sale_contract_auto`

### Models (13 database + 4 transient)

| Model | Type | Description |
|-------|------|-------------|
| `investment.club` | New | Club entity, groups projects and members |
| `investment.project` | New | Investment project with return config (grace period, return amounts, percentages, duration) |
| `investment.membership` | New | Investor membership with partner, fees, renewal, contracts, investor code |
| `membership.renewal` | New | Tracks membership renewals with expiry dates |
| `investment.subscription` | New | Investment subscription linking member to project shares with return tracking |
| `investment.actual.return` | New | Actual return payments (one-time return_1 or periodic return_2) |
| `res.partner` | Inherit | Adds employee/customer/vendor/investor code fields via `contact.py` and `contact_codes.py` |
| `crm.lead` | Inherit | Adds interested project and club fields |
| `res.config.settings` | Inherit | 13 config parameters for membership, payment journals, notifications, approvals |
| `account.move` | Inherit | Links to membership via investor code |
| `account.payment` | Inherit | Links to investment subscription |
| `sale.contract` | Inherit | Links to investment subscription and membership |
| `investment.subscription.reject.wizard` | Transient | Rejection reason wizard |
| `membership.terminate.wizard` | Transient | Membership termination with refund calculation |
| `subscription.terminate.wizard` | Transient | Subscription termination with refund calculation |
| `investor.death.wizard` | Transient | Investor death case handling (transfer or terminate) |

### Key Relationships

```
Club → Project → Subscription ← Membership (Partner)
                    ↓
             Actual Return
```

- **Club** has many **Projects** and **Memberships**
- **Membership** has many **Subscriptions** (investments in projects)
- **Subscription** has many **Actual Returns** (return_1 one-time, return_2 periodic)
- **Membership** has many **Renewals**

### Workflows

**Membership State:** draft → reviewed → initial_invoiced → active → expired/terminated/cancelled
**Subscription State:** draft → reviewed → pending_approval → approved/rejected → paid → active → closed/terminated/cancelled
**Return State:** draft → paid → cancelled

### Key Features

- Investor codes auto-generated by club
- Per-club membership sequences
- Return 1 (one-time fixed amount) and Return 2 (periodic amount or percentage)
- Grace period system (default 3 months) before returns start
- Membership auto-renewal and expiration cron jobs
- Approval workflow for investments
- Contract integration with `sale_contract_auto` module
- Death case handling: transfer to heir or terminate and refund
- Membership termination with financial calculation (deductions, company income)

### Security

`security/ir.model.access.csv` and `security/investment_club_security.xml` define access controls.

### Reports

- Membership contract printing
- Investment contract printing

### Views (14 XML files)

Form/tree/search/kanban views for all models, dashboard, and inherit views for CRM leads, partners, invoices, payments, contracts.

### Cron Jobs

- `_cron_send_renewal_reminders` - Membership renewal notifications
- `_cron_auto_expire_memberships` - Auto-expire past-due memberships
- `_cron_send_payment_notifications` - Return payment notifications

**Author:** Ibrahim Elmasry
**Dependencies:** `base`, `sale_management`, `account`, `portal`, `mail`

| Model | Type | Description |
|-------|------|-------------|

```
draft → confirmed → financial_approved → legal_approved → finished
```

## stock_split_transfer

Splits internal stock transfers into two legs (Source → Transit → Destination) with per-user location visibility.

**Author:** Ibrahim Elmasry
**Dependencies:** `base`, `stock`

### Models (2 inherit)

| Model | Type | Description |
|-------|------|-------------|
| `res.users` | Inherit | Adds `allowed_location_ids` - which locations user can manage |
| `stock.picking` | Inherit | Adds split transfer logic on confirm/validate |

### How It Works

1. User confirms an internal transfer → system detects it needs splitting
2. Creates a transit location if none exists
3. Leg 1: Source → Transit (visible to source location users)
4. Leg 2: Transit → Destination (visible after leg 1 is validated)
5. Each leg visible only to users with access to its respective location
6. Leg 2 auto-confirms when leg 1 is validated

### Fields Added

- `split_parent_id` / `split_child_id` - Links between transfer legs
- `split_visible` - Controls second leg visibility
- `is_split_transfer` - Computed field to identify split transfers

### Views

User form (allowed locations), picking form/list/search (split fields/filters).

### i18n

Arabic translation.

## pos_get_weight_button

Adds a "Get Weight" button to POS control buttons that fetches weight from a scale API.

**Author:** Ibrahim Elmasry
**Dependencies:** `point_of_sale`

### JS

Patches `ControlButtons`:
- Adds `onClickGetWeight()` button
- Fetches from `http://localhost:8000/api/weight`
- Sets selected order line quantity to returned weight
- Handles errors (no selection, invalid weight, service unavailable)

### XML

Extends `ControlButtons` template with a "Get Weight" button.

## pos_refund_restriction

Restricts POS refunds to managers only.

**Author:** Ibrahim Elmasry
**Dependencies:** `point_of_sale`

### JS

- Patches `TicketScreen`: blocks refund UI for non-managers
- Patches `PosOrderline.set_quantity()`: blocks negative quantities for non-managers

### Behavior

- Non-manager cashiers: refund action hidden, negative qty rejected, access denied message shown
- Managers: full refund capability

## sale_kitchen_receipt

### Models (1 inherit)

| Model | Type | Description |
|-------|------|-------------|
| `sale.order` | Inherit | Adds fields: `x_building_floor`, `x_landmark`, `x_driver_name` |
